1. 索引组织表

InnoDB存储引擎的表是根据主键顺序组织存放的，这种存储方式的表被称为 索引组织表(index organized table)。
每张 InnoDB 表都有个主键(primary key),若在建表时未指定主键，则存储引擎会按如下方式指定主键：

  • 若表中有非空的唯一索引(unique not null), 则该列即为主键
    若表中有多个非空唯一索引，则第一个定义的非空唯一索引为主键
  CREATE TABLE `z` (
  `a` int NOT NULL,
  `b` int DEFAULT NULL,
  `c` int NOT NULL,
  `d` int NOT NULL,
  UNIQUE KEY `d` (`d`),
  UNIQUE KEY `c` (`c`),
  UNIQUE KEY `b` (`b`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci

mysql root@127.0.0.1:test> insert into z select 5, 6, 7, 8;
Query OK, 1 row affected
Time: 0.003s
mysql root@127.0.0.1:test> insert into z select 9, 10, 11, 12;
Query OK, 1 row affected
Time: 0.002s

mysql root@127.0.0.1:test> select a, b, c, d, _rowid from z;
+---+----+----+----+--------+
| a | b  | c  | d  | _rowid |
+---+----+----+----+--------+
| 5 | 6  | 7  | 8  | 8      |
| 9 | 10 | 11 | 12 | 12     |
+---+----+----+----+--------+

2 rows in set
Time: 0.013s

上面示例创建的表z有a,b,c,d四个列，b、c、d三列上都有唯一索引，不同的是b列允许NULL；
由于未显式地定义主键，因此会选择非空的唯一索引，_rowid显示表的主键；
虽然c、d列都是非空唯一索引，但d列被首先定义为唯一索引，帮d被视为主键。

_rowid 只能用于查看单个列为主键的情况，对于多列组成的主键无效。

CREATE TABLE `a` (
  `a` int NOT NULL,
  `b` int NOT NULL,
  PRIMARY KEY (`a`,`b`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci

mysql root@127.0.0.1:test> insert into a select 1,1;
Query OK, 1 row affected
Time: 0.003s
mysql root@127.0.0.1:test> select a, _rowid from a;
(1054, "Unknown column '_rowid' in 'field list'")
  • 若不符合上述条件，则 InnoDB 存储引擎会自动创建一个6字节大小的指针

2. InnoDB 逻辑存储结构

从 InnoDB 存储引擎的逻辑存储结构来看，所有数据都被逻辑地存放在一个空间中，即表空间(tablespace)；
表空间又由段(segment)、区(extent)、页(page)组成。
页在一些文档中有时也被称为块(block)。

2.1 表空间

表空间是 InnoDB 存储引擎逻辑结构的最高层，所有数据都在表空间中。
默认情况下有一个共享表空间ibdata1, 但基指定了参数 innodb_file_per_table,则每张表内的数据会单独放到一个表空间内；但每张表的表空间存储的只是数据、索引和插入缓冲Bitmap页；
其它的数据, 如回滚(undo)信息，插入缓冲索引页，系统事务信息，二次定缓冲(double write buffer)等还是在原来的共享表空间中。

InnoDB 存储引擎不会在执行 rollback 时去收缩默认表空间，但是会判断这些undo信息是否还需要，若不需要，
则会将这些空间标记为可用空间，供下次undo使用。

2.2 段

表空间由段组成，常见的段有:数据段、索引段、回滚段等。
InnoDB存储引擎表是索引组织的(index organized),因此数据即索引，索引即数据；
数据段即为 B+ 树的叶子节点(leaf node segment),
索引段即为 B+ 树的非叶子节点(non-leaf node segment)

InnoDB 存储引擎负责对段的管理，DBA不能对其进行控制。

2.3 区

区由连续页组成，每个区的大小为 1MB； 为了保证区中页的连续性，InnoDB 存储引擎一次从磁盘中申请 4～5个区；
默认的 InnoDB 存储引擎页大小为 16KB，即一个区中共有64个连续的页。

InnoDB 1.0.x 版本开始引入压缩页，可以通过参数 KEY_BLOCK_SIZE 设置为 2K, 4K, 8K; 对应的页的数量为512, 256, 128。

InnoDB 1.2.x 版本新增了参数 innodb_page_size, 可以将默认页的大小设为4K，8K，但页中的数据库是不能压缩的；
此时区中页的数量为256， 128，即无论页的大小怎么变化， 区的大小总是 1MB。

为何启用了参数 innodb_file_per_table 后，创建的表默认大小为96KB(我用的 MySQL 8.0.23, 默认为112K）
在每个段开始时，先用32个页大小的碎片页(fragment page)来存放数据，在这此页被用完后，才申请64个连续的页。
其目的是，对于一些小表或undo这类的段，可以在开始时申请较少的空间。

CREATE TABLE `t1` (
  `col1` int NOT NULL AUTO_INCREMENT,
  `col2` varchar(7000) DEFAULT NULL,
  PRIMARY KEY (`col1`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci

#root@best: /var/lib/mysql/test# ll -h t1.ibd
-rw-r----- 1 mysql mysql 112K 4月   5 22:45 t1.ibd

mysql root@127.0.0.1:test> insert t1 select null, repeat('a', 7000);
Query OK, 1 row affected
Time: 0.003s
mysql root@127.0.0.1:test> insert t1 select null, repeat('a', 7000);
Query OK, 1 row affected
Time: 0.004s
mysql root@127.0.0.1:test> insert t1 select null, repeat('a', 7000);
Query OK, 1 row affected

上述 SQL 创建了t1表，col2字段为VARCHAR（7000），这样保证一个页最多2条记录。
接着插入2行数据，因为所有记录都在一个页中，因此没有非叶子节点；
但若插入第三条记录，就会产生一个非叶子节点；新插入的记录导致了 B+ 树的分裂操作；
接着继续插入60条记录，即表t1中共有63条记录，32个页：

create procedure load_t1(count int unsigned)
begin
declare s int unsigned default 1;
declare c varchar(7000) default repeat('a', 7000);
while s <= count do
insert into t1 select null, c;
set s = s+1;
end while;
end;

mysql root@127.0.0.1:test> call load_t1(60);
Query OK, 1 row affected
Time: 0.114s

+----------+
| count(*) |
+----------+
| 63       |
+----------+
1 row in set
Time: 0.011s

-rw-r----- 1 mysql mysql 608K 4月   5 22:52 t1.ibd

在导入了63条数据后，表空间的大小还是小于1MB，
若再插入一条记录，则表空间将按连续64个页的大小开始增长

mysql root@127.0.0.1:test> insert t1 select null, repeat('a', 7000);

2.4 页

页是 InnoDB 磁盘管理的最小单位。默认每个页大小为16KB。从1.2.x开始，可以通过参数 innodb_page_size 修改页的大小为4K, 8K 16K。
若设置完成，则所有表中页的大小都为 innodb_page_size，不可以再次修改。除非用 mysqldump 导入和导出操作来产生新的库。

2.4.1 常见的页类型

  • 数据页(B+tree Node)
  • undo页(undo log page)
  • 系统页(system page)
  • 事务数据页(transaction system page)
  • 插入缓冲位图页(insert buffer bitmap)
  • 插入缓冲空闲列表页(insert buffer free list)
  • 未压缩的二进制大对象页(uncompressed blog page)
  • 压缩的二进制大对象页(compressed blog page)

2.5 行

InnoDB 存储引擎是面对列的(row-oriented),即数据是按行存放的。
每个页最多允许存储16KB/2-200行记录，即7229行。

3 InnoDB 行记录格式

有两种格式来存放行数据: Compatct 和 Redundant。
Redundant是为兼容旧版本而保留的，在 MySQL 5.1 版本中，默认为 Compact 行格式。

3.2 Compact 行记录格式

Compact 记录头信息

名称 大小(bit) 描述
() 1 未知
() 1 未知
deleted_flag 1 该行是否已被删除
min_rec_flag 1 为1，若该记录是预告被定义为最小的记录
n_owned 4 该记录拥有的记录数
heap_no 13 索引堆中该条记录的排序记录
record_type 3 记录类型: 000表示普通，001为B+树节点指针，010为Infimum, 011为Supremum, 1xx保留
next_record 16 页中下一条记录的相对位置
total 40

最后的部分即为实际每个列的数据。

NULL 不占该部分任何空间，即 NULL 除了占有 NULL 标志位，实际存储不占有任何空间。
每行数据除了用户定义的列，还有两个隐藏列: 事务ID列和回滚指针列，分别为6字节和7字节的大小。
若未定义主键，每行还会增加一个6字节的_rowid列。

CREATE TABLE `mytest2` (
  `t1` varchar(10) DEFAULT NULL,
  `t2` varchar(10) DEFAULT NULL,
  `t3` char(10) DEFAULT NULL,
  `t4` varchar(10) DEFAULT NULL
) ENGINE=InnoDB DEFAULT CHARSET=latin1 ROW_FORMAT=COMPACT

InnoDB 存储引擎在页内容是通过一种链表结构来串连各个行记录的。
无论是 CHAR 类型还是 VARCHAR 类型，在 compact 格式下， NULL 值都不占用任何存储空间。

3.2 Redundant 行记录格式

Redundant 是 MySQL5.0之前 InnoDB的行记录存储方式，MySQL5.0支持 Redundant 是为了兼容之前版本的页格式。

Redundant 行记录格式：
字段长度偏移列表|记录头信息|列1数据|列2数据|…|
